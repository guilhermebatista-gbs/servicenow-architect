# Taxonomy & Metamodel (CSDM applied)

## Layers and tables

| Layer | What it is | Table (class) | Character | Used by processes |
|---|---|---|---|---|
| Business Capability | What the business does (e.g. Payments, Content Management, Portfolio Management) | `cmdb_ci_business_capability` | Abstract, hierarchical (has Parent) | EA / planning |
| Business Application | A logical, end-user-facing solution that delivers capabilities | `cmdb_ci_business_app` | Abstract, no IP, stable, one business owner + one IT owner | EA/APM, Demand, Enhancement — **never** Incident/Change |
| Application Service / Service Instance | A concrete instance per environment and/or region and/or module | `cmdb_ci_service_auto` (base, manual OOB); `cmdb_ci_service_discovered` (Service Mapping, or orgs that standardized manual services there); `cmdb_ci_service_calculated` (tag-based). Match the class your instance standardized on — see `cmdb-tables-cheatsheet.md` | Concrete, has associated infra, dynamic (up/down, monitored) | Incident, Problem, Change, Request, Service Mapping |
| Hardware / Software CIs | Servers, OS, DB instances, middleware, runtimes, software products | `cmdb_ci_server` + subclasses, `cmdb_ci_db_instance` + subclasses, `cmdb_ci_appl_*`, `cmdb_software_product_model` (Product Models, auto-created via SAM), `cmdb_sam_sw_install` | Discovered by Discovery / Service Mapping / SAM / HAM | Dependency graph, Technical Debt |
| TRM Product | Approved technology (standards) | TRM standards-product table (name varies by APM implementation / scoped app — verify in the instance dictionary) | Defined by the EA team | TRM catalog, governance |
| TPM Lifecycle | Discovered technology + end-of-support / end-of-life | TPM technology-lifecycle table (name varies by APM implementation — verify in the instance dictionary) | Generated by a scheduled job from install + lifecycle data | EA Workspace, risk score |
| Service ↔ CI association | Link between an Application Service and infrastructure CIs | `svc_ci_assoc` | Populated by Service Mapping or manually | Dependency graph |

> Note on the TRM/TPM tables: many APM implementations use scoped-application tables (often prefixed, e.g. `n_...`) for the Technology Reference Model and Technology Portfolio Management. Do not hard-code a table name from another instance — confirm the actual class in your instance's dictionary.

## Rule for splitting Application Services

Whenever any one of these is true, create a **separate** Application Service:

1. **Different support team** — a different group receives the incidents/changes.
2. **Isolated infrastructure** — cloud vs on-prem, region A vs B, cluster A vs B.
3. **Different environment** — PROD / UAT / DEV / STG / TEST. **Always separate these.**

Rule of thumb: if the service would justify routing a ticket to a different queue **or** scheduling a change window at a different time, it must be its own service.

## Migrating off a Product Model (a common cleanup)

**Context:** some organizations historically parked applications on a Product Model / "Software Package" table and pointed ITSM at it. That violates CSDM and should be corrected.

### Migration plan, per application

1. **Create the Business Application** (`cmdb_ci_business_app`) — Name, IT owner, business owner, parent Business Capability, lifecycle (Invest/Sustain/Retire), classification = "Business Application".
2. **Create one Application Service per environment** — on the Application Service class your instance standardized on (`cmdb_ci_service_auto` OOB manual; `cmdb_ci_service_discovered` if Service Mapping will own them or that's the org standard) — PROD, UAT, DEV as they exist. Each with: support group, hosting type (On-Prem / Cloud-AWS / Cloud-Azure / Cloud-GCP / SaaS), entry point (URL / LB path), SSO/IdP.
3. **Relate service ↔ infrastructure** via `svc_ci_assoc` — servers, DB instances, middleware. Service Mapping does this automatically once the entry point is configured.
4. **Repoint ITSM references** — Incidents, Problems, Changes, Requests that pointed at the Product Model must be repointed to the correct Application Service (a migration script or Transform Map).
5. **Repoint portfolio references** — Demands and Enhancements point at the Business Application, not the service.
6. **Retire the old record** — mark the Product Model record Retired in its lifecycle; do not delete (preserve history).

### Validate before you migrate

- Scripts / business rules that filter on the old Product-Model table.
- Reports / dashboards that count those records.
- External integrations that reference the old record's `sys_id`.
- Form/list customizations on Incident and related tables.

## Canonical relationships (CSDM)

```
Business Capability  ←[supported by]←  Business Application  ←[runs on]←  Application Service  ←[depends on]←  CI (server, DB, software)
                                                                                        ↑
                                                          TPM Lifecycle  ←[discovered from]←  CI
                                                          TRM Product    ←[is approved]→     Product Model
```

- Business Application → Capability via "supports business capability".
- Business Application → Application Service via "runs on / consumed by" (one BA to many services).
- Application Service → CI via `svc_ci_assoc` (many to many).
- Software CI → TPM Lifecycle (scheduled job).
- TPM → TRM → Technical Debt (scheduled job).

## Metadata hygiene

When a Business Application has no owning organization/division, flag it — a BA with no owner is a metadata weakness and should be prioritized in foundations work. The same applies to a BA with no parent Capability or no lifecycle status.
